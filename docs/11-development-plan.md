## Versioning Strategy

Both repositories follow **Semantic Versioning** (`MAJOR.MINOR.PATCH`) managed entirely through Commitizen and conventional commits.

```
v0.x.x  — Development / pre-release (major_version_zero = true)
v1.0.0  — First stable production release
```

### Commit Types → Version Bumps

| Commit type | Example | Bump |
|---|---|---|
| `fix:` | Bug fix, validation correction | PATCH |
| `feat:` | New endpoint, new page, new component | MINOR |
| `feat!:` or `BREAKING CHANGE:` | Changed API contract, schema change | MAJOR |
| `chore:`, `docs:`, `style:`, `refactor:`, `test:` | No bump — recorded in changelog |

### Daily Workflow

```bash
# 1. Make your changes
# 2. Stage files
git add .

# 3. Commit using commitizen (NEVER use git commit -m directly)
bun run cm          # frontend
cz commit           # backend (once configured)

# 4. Push feature branch
git push origin feat/my-feature

# 5. On merge to main — bump version and generate changelog
cz bump             # auto-increments version, writes CHANGELOG.md
git push && git push --tags
```

> **Rule:** Every commit to either repo must go through `cz` / `bun run cm`.
> Direct `git commit -m "..."` messages are not allowed.
> CHANGELOG.md is never edited by hand — always generated by `cz bump`.

## Definition of Done (per phase)

A phase is complete when:
1. All backend endpoints for the phase respond correctly (manual test via API client or curl)
2. All frontend pages for the phase are functional and match the design system
3. Relevant TC-* test cases pass
4. All changes are committed via `cz commit` with proper conventional commit messages
5. `cz bump` has been run and `CHANGELOG.md` reflects the new version in both repos
6. No TypeScript errors (`tsc --noEmit`) and no lint errors (`biome check`)

## Branch Strategy

```
main          ← stable, always deployable, protected
dev           ← integration branch, all features merge here first
feat/*        ← individual feature branches (e.g. feat/group-membership)
fix/*         ← bug fix branches
chore/*       ← tooling, config, dependency updates
```

Flow:
```
feat/group-membership  →  dev  →  main
                          ↑
                    (PR + review)
```

`cz bump` is run on `main` after each phase merge, not on feature branches.
